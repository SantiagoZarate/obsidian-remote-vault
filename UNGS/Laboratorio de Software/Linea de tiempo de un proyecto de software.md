```timeline
[line-3, body-1]
+ Inicio y <br/>Planificación<br/> De Proyecto
+ **Definir la visión del proyecto**
+ El ciclo de vida del proyecto comienza con una visión clara y aprobada del proyecto, marcando la oportunidad de negocio, aspectos generales de la solución, los usuarios destino, costos iniciales y beneficios.
+ 
+ **Definir el alcance del proyecto**
+ Generalmente definido en el contrato del proyecto, establece que se hará y que no se hará en el proyecto. Esto involucra estimaciones iniciales del proyecto, recursos, tiempo, riesgos, etc.
+ 
+ **Creación de la WBS (Web Breakdown Structure)**
+ El trabajo total del proyecto es dividido y subdividido en paquetes pequeños para asegurar la administración y control.
+  
+ **Estimar la WBS**
+ Tiempo, recursos, asignación por roles y esfuerzo requerido para cada tarea definida en la WBS.
+ 
+ **Desarrollar el calendario**
+ Crear un calendario para el proyecto teniendo en cuenta las estimaciones de la WBS, se podría usar el diagrama de Gantt para visualizar la planificación de tareas.
+ 
+ **Definición de Roles y Responsabilidades**
+ Establecer los roles necesarios para el equipo a cargo del proyecto (PM, Po, Devs, Testers, Analyst, Client, Sponsor, User) y sus respectivas responsabilidades.
+
+ **Conformación del equipo**
+ Formar el equipo asegurando de cubrir todos los roles que se necesitan para completar las tareas de la WBS.
+
+ **Identificación de Riesgos**
+ Identificar riesgos potenciales, ya sean internos o externos.
+
+ **Análisis de Riesgos**
+  analizar los riesgos identificados basados en su probabilidad de ocurrencia y el impacto que representan para determinar el riesgo al que nos exponemos.
+ 
+ **Desarrollo planes de riesgo**
+ Idear planes de mitigación y contingencia para cada riesgo identificado.
+ 
+ **Negociación**
+ Utilizar la matriz de negociación entre el equipo de desarrollo y el PO/Cliente para priorizar variables del proyecto (Recursos, calendario, Funcionalidad), con una sola variable fija, la segunda se elije dentro de las posibilidades y la tercera se adecúa.
+ 
+ **Planear versiones del proyecto y entregables**
+ Definir que funcionalidades específicas o incrementos van a ser entregados en diferentes versiones o fases del proyecto.
+ 
+ **Validación**
+ Obtener la validación y el aprobamiento del plan de proyecto de todos los stakeholder relevantes.
```

```timeline
[line-3, body-1, active-color-text-accent]
+ Ejecución y <br/> Monitoreo
+ **Ejecución de tareas y producción de entregrables**
+ El equipo realiza el trabajo pactado en el plan del proyecto
+ 
+ **Monitoreo y control del trabajo del proyecto**
+ Seguimiento de las actividades, monitoreo de la utilización de recursos y comparación del avance actual con el avance planeado
+ 
+ **Reuniones Diarias de Scrum**
+ En metodologías ágiles el equipo se reune de manera diaria por 15 minutos para hacer un repaso del progreso, impedimentos y planear las tareas diarias
+ 
+ **Evaluar el nivel de calidad**
+ Evular la calidad del desarrollo de las funcionalidades basado en su estado (No comenzando, En desarrollo, Completado, en QA, Publicado)
+ 
+ **Usar gráficos Burndown**
+ Seguimiento del trabajo restante para visualizar el progreso y predicir la completitu del proyecto
+ 
+ **Monitorear evolución de Bugs**
+ Seguimiento del ciclo de vida de los defectos (Nuevo, Asignado, Testing, Arreglado, Cerrado, Re-abierto)
+ 
+ **Medir cobertura de Tests**
+ Monitorear hasta que punto el software ha sido testeado comparando lo planeado, lo disponible, pruebas ejecutas y pruebas satisfactorias.
+ 
+ **Aplicar metricas ágiles**
+ Utilizar métricas como la velocidad del equipo, para entender la capacidad de trabajo del equipo y planear futuras iteraciones.
+
+ **Implementación práctivas DevOps**
+ Utilizar una metodología de desarrollo y un conjunto de buenas prácticas enfocadas en la comunicación, colaboración e integración entre el equipo de desarrollo y el equipo de operaciones, incluyendo integración continua y entrega continua.
+ 
+ **Gestión de cambios**
+ Implementar un proceso de control de cambios formal para manejar cualquier modificación al alcance, fecha o presupuesto que pueda sufrir el proyecto.
+ 
+ **Monitoreo de riesgos y control**
+ Continuamente monitorear los riesgos identificados, hacer un seguimiento de su estado.
+
+ **Sprint Review**
+ Al final de cada sprint, el equipo presenta el incremento a los stakeholders, recolecta feedback y se planea la siguiente sprint.

```
